# Cinema Booking System

A complete Cinema Booking System built with **ASP.NET Core Web API** and **Entity Framework Core (Code First)** using **N-Tier Architecture**.

---

## 📐 Architecture - N-Tier Design

This project follows **N-Tier Architecture** to achieve clear separation of concerns and maintainability.

### **Why N-Tier Architecture?**

1. **Separation of Concerns** - Each layer has a single, well-defined responsibility
2. **Maintainability** - Changes in one layer don't affect others
3. **Testability** - Each layer can be tested independently
4. **Scalability** - Easy to scale individual layers
5. **Clean Code** - Easy to understand and explain in interviews

## 📦 Layer Responsibilities

### **1️⃣ Core Layer** (Cinema.Core)

**Purpose**: Contains business logic and domain models

**Contains**:
- ✅ Entities (Movie, ShowTime, Ticket, User)
- ✅ Enums (TicketStatus, UserRole)
- ✅ Interfaces (Repository contracts)

**Does NOT contain**:
- ❌ Entity Framework
- ❌ Database logic
- ❌ HTTP concerns
- ❌ External dependencies

**Why keep it clean?**
- Can be reused in other projects
- Easy to test
- Independent of technology choices
- Core business rules in one place

---

### **2️⃣ Infrastructure Layer** (Cinema.Infrastructure)

**Purpose**: Handles data persistence and external services

**Contains**:
- ✅ ApplicationDbContext (EF Core)
- ✅ Repository implementations
- ✅ Migrations
- ✅ Database-specific logic

**Responsibilities**:
- Communicates with SQL Server
- Implements Core interfaces
- Manages data access

**Why separate from Core?**
- Can switch databases without changing Core
- Database implementation details hidden
- Easy to mock for testing

### **3️⃣ API Layer** (Cinema.API)

## 🗄️ Database Design (Code First)

### **Tables**:
- **Movies**: Movie information
- **ShowTimes**: Movie screening times
- **Tickets**: User ticket bookings
- **Users**: User accounts (Authentication)

### **Relationships**:
```
Movie (1) ──→ (Many) ShowTime
ShowTime (1) ──→ (Many) Ticket
User (1) ──→ (Many) Ticket
```

### **Primary Keys**:
- All IDs are **auto-incremented**
- Generated by the database
- Never sent by the user

## 🔐 Authentication & Authorization

### **Authentication Flow**:

1. **Register**:
   - User provides: Email, Password, Full Name, Role
   - System hashes password using **PBKDF2**
   - Stores user in database
   - Returns JWT token

2. **Login**:
   - User provides: Email, Password
   - System verifies password hash
   - Generates JWT token with claims:
     - User ID
     - Email
     - Full Name
     - Role (User/Admin)
   - Token expires in 24 hours

3. **Authorization**:
   - Client sends token in `Authorization: Bearer {token}` header
   - API validates token signature and expiration
   - Extracts user role and ID from token claims
   - Allows/denies access based on role

## 👥 Roles & Permissions

### **User Role** (Regular User)

**Can**:
- ✅ Register and Login
- ✅ View all movies
- ✅ View showtimes
- ✅ Book tickets
- ✅ Cancel own tickets
- ✅ View own tickets

**Cannot**:
- ❌ Manage movies
- ❌ Manage showtimes

---

### **Admin Role**

**Can**:
- ✅ Register and Login
- ✅ Add/Update/Delete movies
- ✅ Add/Update/Delete showtimes
- ✅ View all data

**Cannot**:
- ❌ Book tickets

## 🔑 Authorization Implementation

### **Controller-Level Authorization**:

```csharp
[Authorize(Roles = "Admin")]
public class MoviesController : ControllerBase
{
    // Only Admins can access
}
```

### **Action-Level Authorization**:

```csharp
[Authorize]  // Any authenticated user
public async Task<IActionResult> GetAll() { }

[Authorize(Roles = "Admin")]  // Only Admins
public async Task<IActionResult> Create() { }
```

### **Allow Anonymous**:

```csharp
[AllowAnonymous]
public async Task<IActionResult> Register() { }
```

## 🔄 Data Flow Explanation

### **Example: Booking a Ticket**

```
1. Client (Postman/Browser)
   ↓
2. HTTP POST /api/tickets
   Headers: Authorization: Bearer {JWT_TOKEN}
   Body: { showTimeId: 1, seatNumber: 10 }
   ↓
3. API Layer (TicketsController)
   - Validates JWT token
   - Extracts user ID from token claims
   - Validates input data
   ↓
4. Infrastructure Layer (TicketRepository)
   - Checks if showtime exists
   - Checks if seats available
   - Creates ticket entity
   - Saves to database using EF Core
   ↓
5. Database (SQL Server)
   - Inserts ticket record
   - Updates available seats
   - Returns saved ticket
   ↓
6. Response back to Client
   - HTTP 201 Created
   - Returns ticket details
```

## 🔧 Setup Instructions

### **1. Prerequisites**:
- .NET 9.0 SDK
- SQL Server (LocalDB or Express)
- Visual Studio / VS Code / Rider

### **2. Update Connection String**:

Edit `Cinema.API/appsettings.json`:

```json
"ConnectionStrings": {
  "DefaultConnection": "Server=YOUR_SERVER;Database=CinemaDb;Trusted_Connection=True;TrustServerCertificate=True"
}
```

### **3. Apply Migrations**:

```bash
cd Cinema.API
dotnet ef database update --project ../Cinema.Infrastructure/Cinema.Infrastructure.csproj --startup-project Cinema.API.csproj
```

### **4. Run the API**:

```bash
cd Cinema.API
dotnet run
```

### **5. Access Swagger**:

```
https://localhost:5001/swagger
```

## 📡 API Endpoints

### **Authentication**:
```
POST /api/auth/register  - Register new user
POST /api/auth/login     - Login user
```

### **Movies** (Admin only for POST/PUT/DELETE):
```
GET    /api/movies       - Get all movies
GET    /api/movies/{id}  - Get movie by ID
POST   /api/movies       - Add movie (Admin)
PUT    /api/movies/{id}  - Update movie (Admin)
DELETE /api/movies/{id}  - Delete movie (Admin)
```

### **ShowTimes** (Admin only for POST/PUT/DELETE):
```
GET    /api/showtimes              - Get all showtimes
GET    /api/showtimes/{id}         - Get showtime by ID
GET    /api/showtimes/movie/{id}   - Get showtimes by movie
POST   /api/showtimes              - Add showtime (Admin)
PUT    /api/showtimes/{id}         - Update showtime (Admin)
DELETE /api/showtimes/{id}         - Delete showtime (Admin)
```

### **Tickets** (User role required):
```
GET    /api/tickets/my-tickets     - Get user's tickets
GET    /api/tickets/{id}           - Get ticket by ID
POST   /api/tickets                - Book ticket
POST   /api/tickets/{id}/cancel    - Cancel ticket
```

## 🧠 Key Concepts for Interviews

### **Why N-Tier Architecture?**

"I chose N-Tier architecture because:
1. **Separation of concerns** - Each layer has one responsibility
2. **Maintainability** - Easy to locate and fix bugs
3. **Testability** - Can test each layer independently
4. **Scalability** - Can scale layers separately
5. **Team collaboration** - Different developers can work on different layers"

---

### **Why Core Layer is Clean?**

"The Core layer contains only business logic and entities:
- No database dependencies (no EF Core)
- No HTTP concerns (no controllers)
- No external libraries
- This makes it reusable, portable, and easy to test"

---

### **How Authentication Works?**

"Authentication flow:
1. User registers → Password is hashed with PBKDF2 (10,000 iterations)
2. User logs in → System verifies password hash
3. If valid → Generate JWT token with user claims (ID, Email, Role)
4. Client stores token → Sends it in Authorization header for each request
5. API validates token → Checks signature, expiration, and extracts user info"

---

### **How Authorization Works?**

"Authorization is role-based:
- JWT token contains user role as a claim
- Controllers/Actions decorated with `[Authorize(Roles = "Admin")]`
- ASP.NET Core middleware validates token and checks role
- If role matches → Request proceeds
- If not → 403 Forbidden response"

---

### **How Data Flows?**

"Data flow example (Booking ticket):
1. Client sends HTTP request with JWT token
2. API layer (Controller) validates token and extracts user ID
3. Controller calls Repository method
4. Repository uses EF Core to interact with database
5. Database saves ticket and returns result
6. Repository returns entity to Controller
7. Controller converts to DTO and sends HTTP response"

---

## 🔒 Security Features

1. **Password Hashing**: PBKDF2 with 10,000 iterations
2. **JWT Authentication**: Signed tokens with expiration
3. **Role-Based Authorization**: Admin vs User roles
4. **SQL Injection Protection**: EF Core parameterized queries
5. **HTTPS**: Secure communication
